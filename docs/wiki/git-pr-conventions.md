## 目的
将代码提交与 PR（Pull Request）写作约束固化为可复用规范，保证每次变更都可审查、可验证、可回滚。

## 适用范围
- 本仓库所有分支的代码/文档变更
- 所有合并到 `main` 的 PR

## Commit 信息规范
- 一个 commit 只做一类事情，避免“顺手”把无关改动混进去。
- 提交信息格式：
  - `<type>: <一句话描述>`
  - 示例：`docs: 记录 PR 闭环流程`
  - 示例：`feat: 新增询价线索导出`
  - 示例：`fix: 修复登录态失效`
- 推荐 type：
  - `feat`：新增功能
  - `fix`：修复缺陷
  - `docs`：文档变更
  - `refactor`：重构（不改变外部行为）
  - `test`：测试相关
  - `chore`：杂项（构建/脚本/CI/依赖等）
- 提交前检查：

```bash
git status -sb
git --no-pager diff
```

## PR 描述规范
PR 描述必须能让审查者在不翻大量上下文的情况下完成决策：是否能合并、是否需要补测、回滚成本有多大。

### 必填内容（模板）

```md
## 变更摘要
（做了什么、为什么做，2-3 句话）

## 变更内容
- （列出关键点与涉及文件/模块）

## 影响范围
- （影响的页面/API/模块/数据/权限）

## 验证方式
- （给出可复现命令与结果，或手工验证路径与截图）

## 回滚方式
- （如何撤回本次变更：回滚 commit、关闭开关、恢复配置/数据等）
```

### 质量门槛
- PR 必须可独立合并：不依赖未合并的隐藏前置改动。
- PR 必须可验证：至少提供一条可复现的验证命令或清晰的手工验证路径。
- PR 必须可回滚：明确回滚动作与影响。

## 合并策略
- 默认：只创建 PR 并交付链接，由你审核与合并；未明确要求不得代你合并。

## 合并后收尾（必须）
当相关需求已开发完成并合并到 `main` 后，必须同步更新“流程标志”，保证需求与交付证据可追溯：
- 需求卡片（`docs/project/features/*.md`）：状态改为 `done`，勾选验收标准，并补充“交付记录”（PR 链接 + 验证记录）。
- Backlog（`docs/project/backlog.md`）：对应条目标记为完成。
- 验证记录要求：除历史遗留外，后续所有标记为 `done` 的需求，验证记录不得留空，必须包含可复现命令与结果要点。
