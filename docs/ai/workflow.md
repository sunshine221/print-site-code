# 稳定 AI 开发流程（Solo 版）

## 0. 核心原则
- AI 负责生成覆盖与收敛不确定性，你负责范围取舍与最终验收。
- 每次对话都必须产出“可落库的文档”或“可验证的代码变更”，禁止停留在口头讨论。
- 任何 AI 输出都必须满足：可测试、可回滚、可解释。
- 开发过程中产生的经验与踩坑必须即时记录到 `docs/wiki/`，并及时更新为可复用流程。
- 涉及“提交到 GitHub/提交本次变更”时，默认只创建 PR 并把 PR 链接交给你审核与合并；未明确要求不得代你合并。

## 1. 角色切换机制（强约束）
在每次开始一个阶段时，先用一句话显式指定角色与产出物，例如：
- “你现在是【需求分析师】，只输出澄清问题清单 + 验收标准 + 边界用例。”
- “你现在是【架构/方案评审】，输出 2-3 套方案对比 + 风险点 + 推荐与理由。”
- “你现在是【实现工程师】，严格按 Tech Spec 改代码，并补齐验证方式。”
- “你现在是【测试工程师】，基于改动生成白盒/黑盒用例并执行；构建并启动页面，交付我做端到端验收。”
- “你现在是【代码审查者】，只做 Review：风险、边界、测试缺口、可维护性。”
- “你现在是【发布负责人】，只输出发布检查单 + 回滚步骤 + 监控点。”

## 2. 阶段与必交付物

### 2.1 需求阶段（需求分析师）
输入：
- 需求一句话 + 背景
- 目标用户/场景
- 不做什么（Out of scope）
- 现状入口（相关页面/API/文件路径）
输出（必须全部具备）：
- 交互规则：每次只问 1 个澄清问题（给选项），我回答后再问下一题；最后整理为澄清问题清单
- 澄清问题清单（按优先级排序）
- 验收标准（checklist 或 Given/When/Then）
- 边界/反例用例（异常、权限、并发、幂等、空值、超限）
落库：
- `docs/project/features/FE-xxx-*.md` 或 `AD-xxx-*.md`（如果没有就先创建一张需求卡片）

### 2.2 方案阶段（架构/方案评审）
输入：
- 已确认的需求卡片链接
- 约束（性能/安全/兼容/上线窗口）
输出：
- 2-3 套方案对比：改动面/风险/扩展性/回滚复杂度
- API 与数据模型漏洞扫描：幂等性、并发、权限、错误码、兼容性
- 推荐方案与关键决策（必须能写成明确约束）
落库：
- `docs/project/specs/<feature>.md`（Tech Spec）

### 2.3 实现阶段（实现工程师）
输入：
- Tech Spec（包含接口契约、数据模型、边界与验收）
输出：
- 变更前：将修改/新增的文件清单 + 测试计划
- 变更后：改动摘要 + 影响面 + 验证命令与结果
强制约束：
- 严禁“凭空引入库”；使用前必须确认仓库已依赖。
- 每次只做一个可回滚的小步（一个模块或一个端点）。
- 实现完成必须进入测试阶段；未通过测试与端到端验收不得提交 PR。

### 2.4 测试阶段（测试工程师）
输入：
- 本次变更的 diff 或文件列表
- Tech Spec 的验收标准与边界用例
输出：
- 白盒/黑盒测试用例清单（覆盖边界）+ 执行结果
- 自动化验证命令与结果（如 lint/build/unit/integration）
- 构建与启动方式 + 访问地址（交付我做端到端黑盒验收）
- 未通过项与修复建议（必要时回到实现阶段）

### 2.5 评审阶段（代码审查者）
输入：
- 本次变更的 diff 或文件列表
输出：
- 关键风险点（安全/权限/并发/幂等/异常处理）
- 测试缺口与补测建议
- 维护性建议（但不大改架构）

### 2.6 发布阶段（发布负责人）
输入：
- 本次变更摘要 + 影响模块
输出：
- 发布检查单（配置、依赖、迁移、开关、数据备份）
- 回滚步骤（代码、数据库、配置、文件）
- 监控点与告警建议（错误率、延迟、关键业务指标）

## 3. 推荐目录（与 PRD 形成闭环）
如果你想把 PRD 的“需求卡片”概念真正落地，建议按下面结构维护：
- `docs/project/backlog.md`
- `docs/project/features/`（需求卡片）
- `docs/project/specs/`（Tech Spec）
- `docs/project/release/`（发布记录与检查单）

## 4. PR 闭环（从改动到可合并）
目标：把一次变更收敛为一个可审查、可验证、可回滚的 PR，并把合并决策交回给你。

### 4.1 改动前：明确边界与可验证性
- 确认本次变更的目标、影响范围与回滚方式。
- 确认目标分支（通常是 `origin/main`）与工作分支（例如 `chore/xxx`）。
- 先看差异再动手，避免改错位置：

```bash
git --no-pager diff origin/main...HEAD
```

### 4.2 改动后：自检与收敛
- 保持变更尽可能小且聚焦：一个 PR 解决一个问题。
- 自检要给出可复现命令与结果（lint/build/test 或手工验证路径）。
- 复核待提交文件与差异：

```bash
git status -sb
git --no-pager diff
```

### 4.3 提交：形成最小可回滚单元
- 暂存与提交：

```bash
git add -A
git commit -m "chore: <一句话描述>"
```

### 4.4 推送：只推送分支，不直接动 main
- 推送到远端分支：

```bash
git push -u origin <branch>
```

### 4.5 创建 PR：把审查信息写全
- 默认策略：只创建 PR 并交付链接给你审核与合并，未明确要求不得代你合并。
- 提交信息与 PR 描述规范：`docs/wiki/git-pr-conventions.md`
- PR 描述建议包含：
  - 本次变更摘要（做了什么、为什么做）
  - 影响面与风险点（权限/兼容/数据/边界）
  - 验证方式与结果（命令、截图/路径）
  - 回滚方式（开关/版本回退/恢复步骤）

### 4.6 交付：把控制权交回给你
- 输出 PR 链接，并等待你确认是否需要补测、补充说明或合并。
