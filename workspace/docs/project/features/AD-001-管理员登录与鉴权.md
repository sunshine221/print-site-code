## 背景与目标
- 背景：后台管理涉及产品、询价线索与站点设置，必须有管理员登录与鉴权保护。
- 目标用户/使用场景：管理员登录后台并保持登录态，未登录用户不能访问 `/admin/**`。

## 范围
- In scope：
  - 管理员登录页：`/admin/login`，用户名（username）+密码登录。
  - 登录接口：`POST /api/auth/login`，成功返回 JWT token + user。
  - 登录态保持：前端持久化 token（localStorage）并在请求中带 Authorization。
  - 路由保护：`/admin` 下所有页面必须鉴权，未登录跳转登录页。
  - 用户信息接口：`GET /api/auth/me`（用于刷新/校验登录态）。
- Out of scope：
  - 多角色权限体系（默认单管理员或少量管理员）。
  - 忘记密码（可选增强）。

## 用户故事
- 作为管理员，我想用用户名+密码登录后台，并在后续访问中保持登录态，从而可以持续维护内容与跟进线索。

## 验收标准（必须可测试）
- [ ] 未登录访问 `/admin` 会跳转到 `/admin/login`。
- [ ] 登录成功后进入 `/admin/dashboard`，刷新页面登录态仍然存在。
- [ ] 登录失败时提示“账号或密码错误”或等价错误信息，不泄露更多细节。
- [ ] 任一后台 API 在未授权时返回 401，前端收到后清理登录态并跳转登录页。

## 边界与反例用例
- token 过期/无效：调用后台接口返回 401，前端必须能自动退登。
- 本地存储损坏：localStorage 内容不是合法 JSON 时也能安全降级为未登录。
- 安全：禁止在日志/错误信息中输出密码与 token。

## 依赖与约束
- 相关入口：
  - 前端登录页：`workspace/src/pages/admin/Login.tsx`
  - 路由保护：`workspace/src/pages/admin/RequireAdmin.tsx`、`workspace/src/App.tsx`
  - 登录态：`workspace/src/stores/auth.ts`
  - 后端鉴权：`workspace/api/auth.ts`、`workspace/api/routes/auth.ts`
  - 环境变量：`JWT_SECRET`、`ADMIN_EMAIL`、`ADMIN_USERNAME`、`ADMIN_PASSWORD`（见 `workspace/.env.example`）

## 实现拆分（最小可回滚）
- 子任务 1：登录页与登录态存储（token/user）。
- 子任务 2：请求层统一注入 Authorization（如 `apiFetch`）。
- 子任务 3：路由与接口 401 处理（自动退登与跳转）。
- 子任务 4：`/api/auth/me` 用于启动时校验登录态（可选增强）。

## 验证方式
- 自动化：`npm run check`、`npm run lint`、`npm run build`
- 手动路径：
  - 访问 `/admin`，确认跳转到 `/admin/login`
  - 使用正确/错误密码登录，验证成功/失败提示
  - 登录后刷新页面，确认仍登录
  - 手动清空 token 或篡改 token，访问后台页应退登

## 发布与回滚
- 发布检查点：登录成功率、401 退登逻辑、后台路由保护。
- 回滚方式：前后端鉴权逻辑需一致回滚；如涉及数据库管理员表结构变更需配合 DB 备份。
